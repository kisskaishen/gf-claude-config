# 性能审计

1. **先测量。** 运行生产构建并检查包组成/体积；与之前的基线对比。基于数据优化，而非凭感觉。
2. **加载** — 重型/不常用的组件是否懒加载？路由是否分割？是否有大量新依赖可以用更小的工具函数或原生 API 替代？是否有命名空间全量导入破坏了 tree-shaking？
3. **渲染** — `v-for` 是否用了稳定的 id 作 key？长列表是否虚拟化？`watch` 是否做了 `computed` 该做的事？渲染中是否有昂贵的计算？高频事件处理器是否防抖/节流？
4. **响应式** — 大型数据结构是否存在意外的深层响应式；不必要的克隆。
5. **资源** — 图片尺寸合适且用现代格式，屏幕外的懒加载，预留尺寸以避免布局偏移。
6. **报告** — 按严重程度列出发现，附带 `file:line` 和具体修复方案，尽可能量化（"增加约 Xkb"、"渲染了全部 N 行"）。不推荐过早的微优化。
